# node-to-react-prep

> Автоматизированная подготовка тестовых данных для будущего React-приложения  
> **Тема курсового проекта:** «Система онлайн-бронирования помещений»

> ⚠️ **Важно:** Файл `data/db.json` **не хранится в репозитории** (добавлен в `.gitignore`). Он генерируется автоматически при запуске команды `npm run dev`. Это соответствует лучшим практикам: в репозитории хранится только код, а тестовые данные создаются локально у каждого разработчика.

---

## 🟢 Настройка среды

```json
// package.json
{
  "type": "module"
}
```

**ПОЧЕМУ `"type": "module"` критично для перехода к React?**  
React-экосистема и современный фронтенд-стек используют синтаксис ES-модулей (`import`/`export`). Включение этой настройки на этапе подготовки данных гарантирует, что утилиты генерации, валидации и будущей сборки будут использовать единый стандарт. Это исключает ошибки транспиляции, упрощает переиспользование типов и функций между бэкенд-скриптами и клиентским кодом, а также делает миграцию данных в React-проект бесшовной — без необходимости переписывать импорты или настраивать Babel.

## 📜 Скрипты автоматизации

| Команда                 | Описание                                                                   |
| ----------------------- | -------------------------------------------------------------------------- |
| `npm run dev`           | Запуск генератора данных: создаёт `data/db.json` с 50 тестовыми записями   |
| `npm run server`        | Поднимает локальный HTTP-сервер на `localhost:3000` для отдачи `db.json`   |
| `node src/validator.js` | Проверка структуры `db.json` на соответствие требованиям React-компонентов |

## 🎤 Интервью-вопросы (Interview Corner)

### 1. О безопасности: почему мы добавляем `node_modules` в `.gitignore` и не отправляем её на GitHub?

Папка `node_modules` содержит тысячи файлов, включая бинарные зависимости, привязанные к конкретной ОС и архитектуре. Её игнорирование:

- Ускоряет `git push/pull` и уменьшает размер репозитория;
- Предотвращает конфликты версий между разработчиками;
- Гарантирует воспроизводимость сборки: каждый участник команды устанавливает зависимости заново через `package-lock.json`, а не копирует чужой кеш.

### 2. О версиях: что значит символ `^` в версии пакета (например, `"@faker-js/faker": "^8.4.1"`)?

`^` (caret) разрешает автоматическое обновление до последней **совместимой** минорной или патч-версии (`8.5.0`, `8.4.2`), но блокирует мажорные релизы (`9.0.0`). Это обеспечивает баланс: проект получает исправления безопасности и багов, но не ломается из-за breaking changes, которые могут появиться в новой мажорной версии.

### 3. О выживании: если ты скачал проект из интернета без папки `node_modules`, какой командой ты его «оживишь»?

`npm install` (или сокращённо `npm i`). Команда читает `package.json`, разрешает дерево зависимостей согласно `package-lock.json`, скачивает точные версии пакетов в локальную папку `node_modules` и подготавливает среду к запуску. Без `package-lock.json` могут быть установлены несовместимые версии — поэтому он тоже коммитится.

### 4. Провокационный вопрос: «А можно ли обойтись без этого пакета, написав всё на чистом JS? В чем будет минус?»

**Да, технически можно.** Но минусы будут существенными:

- **Увеличение объёма кода**: ручная генерация реалистичных имён, дат, цен потребует сотен строк вместо одного вызова `faker.commerce.productName()`;
- **Риск edge-cases**: работа с датами (високосные годы, часовые пояса, форматирование) — источник скрытых багов, которые `date-fns` решает проверенными алгоритмами;
- **Отсутствие типизации и валидации**: без `zod` придётся писать вложенные `if/else` для проверки структуры объектов, что усложняет поддержку и тестирование;
- **Потеря времени на разработку**: пакеты экономят ~40–60% времени на рутинные задачи, позволяя сосредоточиться на бизнес-логике и UX.

> 💡 **Вывод**: использование проверенных библиотек — это не «костыль», а профессиональная практика, которая повышает надёжность, читаемость и скорость разработки.

## 🔍 Tech Stack Research (для курсового проекта)

> Тема: **«Система онлайн-бронирования помещений»**  
> Реализация: SPA на React + localStorage, многошаговая форма, валидация, работа с датами.

### 1. `date-fns`

1. **Название пакета:** `date-fns`
2. **За что отвечает:** Модульная библиотека для парсинга, форматирования и математических операций с датами. Поддерживает tree-shaking — импортируются только используемые функции.
3. **ПОЧЕМУ он нужен в моей теме:** В системе бронирования необходимо генерировать сетку доступных временных слотов, проверять пересечения броней по дням и выводить дату в формате `DD.MM.YYYY HH:mm`. `date-fns` решает эти задачи без создания глобальных объектов-дат, что предотвращает мутации состояния в React и ускоряет рендер компонента `TimeSlotPicker`.

### 2. `react-hook-form`

1. **Название пакета:** `react-hook-form`
2. **За что отвечает:** Высокопроизводительная библиотека для управления формами с минимальным количеством ре-рендеров и встроенной интеграцией с валидаторами.
3. **ПОЧЕМУ он нужен в моей теме:** Проект требует многошаговой формы (регистрация → выбор помещения → выбор слота → подтверждение). `react-hook-form` позволяет собирать данные с каждого шага в единый объект, валидировать поля в реальном времени и предотвращать лишние перерисовки всего SPA, что критично для плавного UX на мобильных устройствах.

### 3. `zod`

1. **Название пакета:** `zod`
2. **За что отвечает:** Схема-ориентированная библиотека для валидации данных с автоматическим выводом типов и строгими проверками структуры объектов.
3. **ПОЧЕМУ он нужен в моей теме:** При сохранении броней в `localStorage` необходимо гарантировать, что объект содержит валидный email, корректный timestamp и обязательные поля. `zod` блокирует запись битых данных на этапе ввода, а при интеграции с `react-hook-form` сокращает код валидации формы до 5–10 строк, исключая ручные `if/else` проверки.

### 4. `lucide-react`

1. **Название пакета:** `lucide-react`
2. **За что отвечает:** Набор современных, оптимизированных SVG-иконок, специально спроектированный для React-проектов с поддержкой tree-shaking.
3. **ПОЧЕМУ он нужен в моей теме:** Для визуального различения типов помещений (`ServiceSelector`), отображения статусов брони (свободно/занято/подтверждено) и индикации шагов многошаговой формы требуются понятные иконки. `lucide-react` позволяет импортировать только нужные компоненты, не раздувая итоговый бандл приложения.

### 5. `nanoid`

1. **Название пакета:** `nanoid`
2. **За что отвечает:** Генерация уникальных, безопасных, компактных ID в формате URL. Работает синхронно и асинхронно, поддерживает кастомный алфавит.
3. **ПОЧЕМУ он нужен в моей теме:** Каждой брони в `localStorage` требуется уникальный идентификатор для последующего поиска, редактирования и удаления. `nanoid` генерирует короткиие ID (по умолчанию 21 символ), которые безопаснее `Math.random()` и удобнее длинных UUID при отображении в интерфейсе.

## 🚀 Инструкция по запуску

```bash
# 1. Клонировать репозиторий и перейти в папку проекта
git clone <your-repo-url>
cd node-to-react-prep

# 2. Установить зависимости
npm install

# 3. Запустить генератор тестовых данных (создаст data/db.json)
npm run dev
# ✅ В папке data/ появится файл db.json

# 4. (Опционально) Запустить локальный сервер для отдачи данных
npm run server
# 🌐 Откройте в браузере: http://localhost:3000/db.json

# 5. (Опционально) Проверить валидацию структуры БД
node src/validator.js
```
